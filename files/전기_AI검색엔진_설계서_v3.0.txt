1. 프로젝트 개요
==================================================

한 줄 정의:
전기 엔지니어링 특화 AI 플랫폼.
글로벌 논문/규격/뉴스 실시간 수집 + 할루시네이션 없는 출처 기반 답변
+ 영수증 계산기 + 날짜 박제 + 협업 + 현장 모바일까지 원스톱 엔지니어링 플랫폼.

- 가트너 예측: 2026년까지 전통 검색엔진 볼륨 25% 감소, AI 버티컬로 이동

- 전기 특화 버티컬 검색: 전 세계 공백 상태

2. 핵심 철학 및 원칙
==================================================

원칙 1: 할루시네이션 제로
- AI = 수집된 문서 기반 요약만 수행
- 출처 없는 답변 = 시스템 레벨에서 차단
- 수치/규격 = 계산 엔진이 직접 연산

원칙 2: 날짜 박제
모든 데이터에 타임스탬프 필수.
원문 발행일 / 수집일 / 검증일 / 규격 버전 / 계산 실행 시각 / 엔진 버전.
"언제 기준인지 모르면 현장에서 못 쓴다."

원칙 3: 영수증 투명성
계산 과정 전체 공개.
출처 논문/규격 조항까지 명시.
AI가 계산했다 X -> 엔진이 직접 돌린 결과.

원칙 4: 도메인 플러그인
전기 -> 기계 -> 토목 -> 화학 -> 전체 공학.
코어 건드리지 않고 도메인만 추가.

원칙 5: BYOK
LLM 비용 = 사용자 부담 (본인 API 키 사용).
플랫폼 비용 = 크롤링 + DB + 검색만.
API 키는 서버에 저장하지 않음.

원칙 6: 법적 방패
엔지니어의 계산서 = 법적 근거 문서.
날짜 박제 + 블록체인 공증 = 당시 현행 규격 기준으로 작성됨을 타임스탬프로 영구 증명.

소스 계층:

  IEEE Xplore - 주 1회
  IEC 공식 문서 - 월 1회
  KEC/전기사업법 - 월 1회
  DOE / EPRI - 주 1회
  arXiv cs.AI/LG - 일 1회

Tier 2 (신뢰):
  Elsevier / Springer - 주 1회
  MDPI - 주 1회
  Hugging Face Papers - 일 1회
  Papers With Code - 일 1회

Tier 3 (참고):
  IEEE Spectrum - 6시간
  Power Magazine - 6시간
  T&D World - 6시간
  EE Times - 6시간
  BloombergNEF - 12시간

Airflow DAG 구성:

DAG 1 논문 크롤링 (주 1회):
  crawl_ieee -> crawl_arxiv -> crawl_elsevier
  -> domain_filter -> dedup_by_doi

8. AI 검색 엔진

검색 파이프라인:
  자연어 질문 입력
  -> 1단계: 질문 분석 (BYOK LLM) - 도메인 분류, 검색 의도, 키워드 추출, 한영 쿼리
  -> 2단계: 하이브리드 검색 (Weaviate) - 벡터 + BM25 + Tier 가중치 + 도메인 필터
  -> 3단계: 지식 그래프 연계 - 연관 개념 탐색, 학습 경로 제안
  -> 4단계: 재랭킹 - Cross-encoder + 최신성 + Tier 점수
  -> 5단계: RAG 답변 (BYOK LLM) - 문서 기반으로만 답변 강제
  -> 6단계: 할루시네이션 검증 - 원문 일치도, 수치 범위, 출처 없는 주장 감지
  -> 7단계: 날짜 메타데이터 부착 - 모든 출처에 수집일/발행일/검증일 표시
  -> 8단계: 최종 출력 - 답변 + 출처(Tier+날짜) + 관련 계산기 + 관련 규격

검색 타입별 처리:

논문 검색: IEEE + arXiv 우선, 2년 이내 가중치 상향, 핵심 수치/결론 추출
규격 검색: Tier 1 소스 우선, 개정 이력 표시, 국내 대응 규격 연결
법령 검색: 전기사업법/기술기준 우선, 조항 번호 명시, 최근 개정 여부 표시
기술 Q&A: 복합 소스 검색, 개념 설명 + 근거 논문, 관련 계산기 추천

  answer: 답변 텍스트
  confidence: 신뢰도 점수 (0~1)
  sources: [{title, source, tier, url, page, relevance, published_at, collected_at, verified_at}]
  related_calculators: [계산기명]
  related_standards: [규격명]
  search_type: 검색 타입
  domain: 도메인

9. 에이전트 워크플로우
==================================================

개념:
  기존 검색: 질문 -> 답변 (끝)
  에이전트: 질문 -> 답변 -> 다음 액션 -> 다음 액션 -> 결과물
  목표: 엔지니어의 업무 흐름을 플랫폼 안에서 완결

워크플로우 예시 (수변전설비 설계):

  입력: "22.9kV 수변전설비 설계해줘 (부하 500kW, 역률 0.9, 케이블 200m)"

  Step 1: 변압기 용량 계산
    -> 영수증 #001 생성 (날짜 박제 포함)
    -> 결과: 667kVA -> 표준 750kVA 선정

  Step 2: 관련 KEC 조항 자동 확인
    -> KEC 210.3, 230조 검색
    -> 2024년판 현행 확인

  Step 3: 케이블 선정 계산
    -> 영수증 #002 생성
    -> 결과: 240mm2 CV 케이블

  Step 4: 전압강하 검증
    -> 영수증 #003 생성
    -> 결과: 2.3% (허용 3% 이내 통과)

  Step 5: 차단기 용량 계산
    -> 영수증 #004 생성

  Step 6: 설계 체크리스트 생성
    통과: 변압기 용량 (KEC 210.3 적합)
    통과: 케이블 선정 (KEC 232 적합)
    통과: 전압강하 (3% 이내)
    통과: 차단기 협조
    요주의: 접지 저항 계산 필요

  Step 7: 설계 보고서 초안 생성
    -> 모든 영수증 통합
    -> PDF 다운로드

에이전트 클래스 구조:
  EngineeringAgent (domain, byok_key)
    - llm: LLMAdapterFactory.create(byok_key)
    - calculator: CalculatorEngine()
    - searcher: RAGSearcher()
    - receipts: []

  run(task) 메서드:
    1. 태스크 분석 -> 단계 계획
    2. 단계별 실행 (calculate / search / verify_standard)
    3. 각 단계 날짜 박제 적용
    4. 결과 통합 -> 체크리스트 + 보고서 초안

영수증 출력 구조:

  [메타 정보 - 날짜 박제]
  계산 일시: 2026.04.04 14:32:11
  적용 규격: KEC 2024년판
  규격 현행: 2026.04.01 확인됨
  엔진 버전: CalcEngine v2.3.1
  무결성 해시: a3f2c1...8d4e

  [적용 공식]
  P = 루트3 x V x I x cosθ
  근거: KEC 210.3 / IEC 60076-1

  [입력값]
  결선 방식: 3상
  선간전압 V: 22,900 V
  전류 I: 100 A
  역률 cosθ: 0.95

  [계산 과정]
  Step 1: 루트3 = 1.7320
  Step 2: 22,900 x 1.7320 = 39,663 V
  Step 3: 39,663 x 100 = 3,966,280
  Step 4: 3,966,280 x 0.95 = 3,767,966
  Step 5: 3,767,966 / 1,000 = 3,767.97 kVA

  [결과]
  계산값: 3,767.97 kVA
  권장 선정: 4,000 kVA (표준 용량)
  주의: 수용률/부등률 적용 검토 필요
  주의: 장래 부하 증설 여유분 고려

  [액션]
  PDF 저장 / 공유 링크 / 재계산 / 블록체인 공증 / 저장 / 코멘트

18. OCR 명판 인식
==================================================

인식 대상:
  변압기 명판 (용량/전압/전류/임피던스/제조일)
  차단기 명판 (정격전류/정격전압/차단용량)
  케이블 레이블 (규격/단면적/제조사)
  전기 도면 상 수치
  규격서 페이지 스캔

처리 흐름:
  카메라 촬영 또는 이미지 업로드
  -> PaddleOCR 텍스트 추출
  -> 전기 도메인 파라미터 파싱 (전압, 전류, 용량, 임피던스)
  -> 파라미터 -> 계산기 자동 매핑
  -> 이 데이터로 변압기 용량 계산할까요? 제안
  -> 사용자 확인 -> 계산 실행 -> 영수증 생성

사용 기술:
  PaddleOCR - 메인 OCR 엔진
  Tesseract - 보조 OCR
  커스텀 파서 - 전기 파라미터 추출

19. 블록체인 공증
==================================================

필요성:
  전기 설계 계산서 법적 보존 기간: 준공 후 10년 이상
  문제: 당시 현행 규격 기준이었나? 위변조 됐나?
  해결: 블록체인 타임스탬프 공증 -> 위변조 불가, 영구 증명

공증 프로세스:
  계산 영수증 완성
  -> SHA-256 해시 생성 (영수증 전체 내용 기반)
  -> IPFS 분산 저장 (CID 생성)
  -> 블록체인 타임스탬프 트랜잭션 (해시 + 시각 기록)
  -> 공증 완료 배지 부착
  -> 누구나 해시로 검증 가능

검증 결과 예시:
  영수증 ID: receipt_abc123
  해시값: a3f2c1...8d4e
  블록: #12345678
  시각: 2026.04.04 14:32:11 KST
  규격: KEC 2024년판
  결과: 검증 완료 - 원본과 일치, 위변조 없음

가치:
  법적 근거: 당시 현행 규격 기준임을 타임스탬프로 증명
  감리 제출: 계산 근거 문서화 자동화
  분쟁 대비: 설계 당시 기준 영구 보존

22. BYOK 구조
==================================================

키 관리 흐름:
  사용자 API 키 입력
  -> 클라이언트에서 AES-256 암호화
  -> 암호화된 키만 서버 전송
  -> DB 저장 (복호화 불가 구조)
  -> API 호출 시: 복호화 -> 헤더 삽입 -> LLM 호출
  -> 응답 후 메모리 즉시 삭제

지원 LLM 어댑터:
  openai -> OpenAIAdapter (GPT-4o)
  anthropic -> AnthropicAdapter (Claude)
  gemini -> GeminiAdapter
  ollama -> OllamaAdapter (로컬 LLM)
  모든 어댑터 공통 인터페이스: query(system, user, context), embed(text)

비용 분담:
  플랫폼 부담: 크롤링, DB, 검색, OCR, 수치 계산
  사용자 BYOK: LLM 답변 생성, 뉴스 번역/요약, 파라미터 추출

27. 정확도 목표

영역별 목표 정확도:

계산기 (영수증): 99.5% 이상
  근거: Python 엔진 직접 연산, AI 계산 아님

AI 검색 (RAG): 85~92%
  근거: 도메인 특화 RAG, 일반 Perplexity 대비 5~10%p 높음

뉴스 번역/요약: 88~93%
  근거: 원문 출처 병기로 검증 가능

자격증/일정: 95% 이상
  근거: 큐넷 공식 데이터 크롤링

OCR 명판 인식: 90% 이상
  근거: 정형 데이터 특성

정확도 향상 전략:

단기:
  청크 크기 A/B 테스트
  전기 도메인 임베딩 파인튜닝
  수치 범위 자동 검증 강화

중기:
  사용자 피드백 루프 (틀린 답변 신고)
  커뮤니티 버그 리포트 반영
  전문가 검수 레이어 추가

장기:
  전기 특화 LLM 파인튜닝
  지식 그래프 기반 정확도 향상
