ESA AI Architecture — eh-universe-web 인프라 구축 계획
Context
ESA (Electronic Search AI) v11.0 설계서 + AI Architecture v1.0의 핵심 인프라를 eh-universe-web 프로젝트에 구축한다.
기존 프로젝트의 인프라(Firebase Auth, Supabase PostgreSQL, Vercel AI SDK, BYOK, Stripe, i18n, NOA 보안, Panel Registry)를 최대한 재활용하면서,
결정론적 전기 엔지니어링 계산 엔진의 5개 레이어를 신규 구현한다.
핵심 원칙: "LLM은 귀(의도 파악), ESA 엔진은 뇌(계산+판정), 리포트는 입(출력)"

Phase 2: Layer 01 — 상수 + 규격 DSL (Day 12)
2-1. 물리 상수 (constants/physical.ts)

2-3. 허용전류 테이블 (standards/ampacity-tables.ts)

2-4. KEC 조건 트리 (standards/kec/*.ts)
KEC 232.52 (전압강하): 간선 3%, 분기 3%, 합산 5%
KEC 212.3 (차단기): 125% 규칙 + 표준 용량 매핑
KEC 142.5 (접지): 접지 저항 10Ω 이하
2-5. 단위 변환 (standards/unit-conversion.ts)
AWG → mm² 고정 매핑 테이블 (AI 추정 금지)
ft ↔ m, °F ↔ °C, psi ↔ kPa

3-3. 입력 검증 (engine/validators.ts)
전압: 100600V (저압), 1kV35kV (고압)
전류: 0.1~10000A

Phase 4: Layer 3 — SJC 판정 시스템 (Day 3)
4-1. 판정 엔진 (sjc/judge.ts)
judge(calcResult, codeArticle) → Judgment
PASS: 모든 조건 충족
HOLD: 입력 부족
FAIL: 규격 위반 (위반 사유 + 근거 조항)
BLOCK: LLM 금지 구역 침범
4-2. 출처 추적 (sjc/source-tracker.ts)
모든 수치에 SourceTag 부착
출처 없는 수치 → BLOCK
[USER] 사용자 입력, [CODE] 규격 DB, [CALC] 엔진 산출, [CONST] 물리 상수
4-3. 영수증 생성 (sjc/receipt.ts)
모든 계산 → Receipt 자동 생성
입력 파라미터, 적용 규격, 공식, 결과, 판정, 타임스탬프
면책 조항 자동 삽입
Phase 5: Layer 4 — LLM 통제 계층 (Day 3~4)
5-1. ESA Tool 정의 (llm/tools.ts)
Vercel AI SDK tool() 형식으로 정의
calculate_voltage_drop, calculate_cable_sizing, calculate_breaker_sizing 등
lookup_code_article, convert_unit, compare_scenarios
각 Tool = 계산 엔진 직접 호출 (LLM 우회)
5-2. 시스템 프롬프트 (llm/system-prompt.ts)
"당신은 전기 엔지니어링 인터페이스입니다.
 당신은 계산하지 않습니다. Tool을 호출합니다.
 Tool 없이 수치를 말하면 시스템이 차단합니다."
5-3. 출력 필터 (llm/output-filter.ts)
LLM 응답 파싱 → 수치 추출 → 출처 태그 확인
"약", "대략", "보통", "일반적으로" + 수치 → 삭제
규격 조항 직접 인용(DB 경유 안 한 것) → 삭제
5-4. 의도 파서 (llm/intent-parser.ts)
다국어 의도 파악 → Tool 매핑
"전선 뭘로 써야 해?" → calculate_cable_sizing
"전압강하 기준이 뭐야?" → lookup_code_article
