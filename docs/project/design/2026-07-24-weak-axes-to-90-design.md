# 약점 축 90점 설계 — 정확도 계량·게이트 안정성·재현성

2026-07-24 · 상태: **설계** · 선행: [토폴로지 스냅·분류 재설계](2026-07-24-topology-snap-and-classification-redesign.md) · 심판: [실증 원장](../../VALIDATION_EVIDENCE.md)

도면 시스템 채점(2026-07-24)에서 약점으로 확인된 세 축을 90 이상으로 올리는
설계다. 현재: 정확도 계량 35 · 게이트 안정성 55 · 재현성 60.

## 채점의 정의부터

**계량 축의 점수는 측정값의 크기가 아니라 측정의 존재와 신뢰성이다.**
15장을 측정해 edgeF1이 0.80이 나와도 계량 축은 90이 될 수 있다 — 신뢰할 수
있는 숫자가 존재하고 제품 주장이 그 숫자에 결박되기 때문이다. 그 경우
`verified95=false`가 유지되는 것이 정확히 옳은 동작이다. 점수를 올리기 위해
임계값을 낮추거나 라벨을 관대하게 만드는 것은 이 설계의 실패 조건이다.

## W1 — 정확도 계량 35 → 90: 라벨 공장

병목은 알고리즘이 아니라 **평가기가 요구하는 형태(공간 bounds 포함)의 실도면
정답이 0장**이라는 것이다. 전문가가 백지에서 라벨을 만들면 도면당 수 시간이
든다. 설계의 핵심은 **제품이 이미 가진 교정 인프라를 라벨 공장으로 재사용**해
전문가 시간을 교정·판정에만 쓰게 하는 것이다.

### 공장 파이프라인 (기존 인프라 재사용)

```
① pre-label   파이프라인 실행 → 기호·결선·문자 초안 (자동, 0 전문가시간)
② 블라인드 교정  이종 모델 2좌석이 상호 무인지로 초안을 교정
               — /api/drawing-jobs/[id]/corrections 로 수행:
               CAS 버전·요청 고유키·전후 재계산 영수증이 그대로 증명이 된다
③ 함정 검증    라벨에 심은 trap(기존 p14 라벨의 trap 관행)을 좌석이
               적발했는지 채점 — 기준 미달 좌석의 교정 전체 기각
④ 판정        두 좌석 불일치 항목 + humanReviewQueue 를 인간 도메인
               전문가가 판정, 표본 ≥20% 스팟체크
⑤ 출고        adjudicated label + SHA-256 + 교정 영수증 체인 첨부
```

②의 요점: pre-label 교정 방식은 앵커링 편향이 있다. 그래서 (a) 이종 모델
2좌석 상호 무인지, (b) trap 경계(③), (c) 인간 판정(④) 세 겹으로 상쇄한다.
trap은 "초안이 맞는 것을 틀렸다고 하는지"가 아니라 **"초안이 틀린 것을
그대로 통과시키는지"** 를 잡는 장치다.

### 독립성 사다리

| 등급 | 정의 | 현재 |
|---|---|---|
| IND-1 | 동일 모델 2좌석 블라인드 | p14 라벨이 여기 |
| IND-2 | **이종 모델** 2좌석 + trap 적발 기준 통과 | 공장 기본 출고 등급 |
| IND-3 | 인간 도메인 전문가가 불일치·humanReviewQueue 판정 + ≥20% 스팟체크 | claim 자격 등급 |

### 표본 크기 — 도면 수가 아니라 항목 수가 목표다

임계값 0.95를 신뢰구간 반폭 0.01로 말하려면 (정규근사, 실게이트는 Wilson):

```
n ≥ 1.96² × 0.95×0.05 / 0.01²  ≈ 1,825 (엣지·기호)
n ≥ 1.96² × 0.98×0.02 / 0.01²  ≈   753 (접점, 임계 0.98)
```

체크인된 실도면 실측 수율이 도면당 엣지 73~314·접점 36~54이므로:

- **목표: 실도면 15장** — 엣지 ≥1,800 · 기호 ≥1,500 · 접점 ≥700
- 층화: 난이도(초/중/고) × 유형(SLD·패널 스케줄·배치) 각 칸 최소 3장
- 항목은 도면 안에서 군집이므로(iid 아님) 보조 지표로 **도면별 최저값**을
  함께 보고한다 — 평균이 좋은데 특정 도면 유형이 무너지는 것을 숨기지 않는다.

### claim 자격 술어 (manifest에서 기계 판정)

`claimEligible=true`는 다음 전부가 참일 때만:

```
strata 충족(3×3 각 칸 ≥3장) ∧ n 충족(위 표) ∧ independence ≥ IND-3
∧ 동일 데이터셋 3회 반복 영수증 ∧ Ed25519 attestation 서명
∧ evaluatorVersion 고정 ∧ 예측 해시 일치
```

전부 기존 부품이다 — fail-closed 게이트(`gate:sld-golden`)·서명 검증·반복
영수증(`runBenchmarkSuite`)은 이미 구현돼 있고, 지금은 넣을 데이터가 없어
의도대로 exit 1일 뿐이다.

### 공급자 매트릭스

`runBenchmarkSuite` × {OpenAI, Gemini, Claude} × 3회 반복. 지표 평균·분산,
도면당 비용, 타임아웃·누락률을 같은 15장에서 기록한다. PROJECT_STATE
'다음 첫 행동 2'와 동일 항목 — 이 설계가 그 실행 계획이다.

### 비용 (정직하게)

전문가 시간: 교정 검토 도면당 1~2시간 + 판정/스팟체크 → **15장에 약 3~4
전문가·일**. 백지 라벨링(도면당 수 시간 × 15장) 대비 약 1/3. 이 시간은
설계로 제거할 수 없다 — 제거하면 IND-3이 무너지고 지금의 IND-1 한계를
재생산한다.

## W2 — 게이트 안정성 55 → 90

1. **M0 실측**으로 `gate:pdf` 6/17의 틀린 쪽(파서 vs fixture)을 확정하고 수리.
2. **실도면 유래 스트로크 재생 fixture 추가** — M0의 남은 가설이 "게이트
   fixture가 실도면을 대표하지 못한다"이므로, 자체 확보 공개 도면(kimm)의
   스트로크 좌표 서브셋(~50선)을 content stream으로 재생하는 R-real 케이스를
   넣어 손제작 fixture와 실도면 사이의 대표성 간극을 게이트 안에서 닫는다.
3. **flake 규율**: 재시도로 green 만들기 금지. flaky 게이트는 DORMANT 대장
   방식의 격리 항목으로만 제외 가능(사유·소유·복귀 조건 기록).
4. **승격 조건**: `live-gates` 14일 연속 green 후 required check로 승격.
   그 전에는 required로 만들지 않는다 — 불안정한 게이트를 필수로 만들면
   우회 관행이 생기고 그게 55점의 원인이다.

## W3 — 재현성 60 → 90

두 번의 드리프트(ts-node 부재, gate:pdf 로컬 17/17 vs CI 6/17)가 원인이다.
공통 뿌리: **개발 머신의 node_modules가 lockfile과 어긋나도 아무것도 막지
않는다.**

1. **환경 동등성 게이트** (`scripts/verify-env-parity.mjs`, enforce.ps1 첫
   단계): node 버전 = `.nvmrc` ∧ `npm ls` 정합(설치 트리 손상 시 비0 종료)
   ∧ lockfile working-tree clean. 실패 시 이후 단계 차단 — "내 머신에서는
   됐다"를 기계적으로 불가능하게 만든다.
2. **`packageManager`·`engines` 고정**: package.json에 npm 버전과 node 범위를
   선언해 corepack/설치기가 강제하게 한다.
3. **CI 정본 규율**: PROJECT_STATE '검증'에 올리는 exit 0은 CI run ID를
   인용한다. 로컬 실행은 참고로만 기록한다. (이 PR에서 이미 실행 중인 관행의
   명문화.)

## 체크포인트와 점수

| 단계 | 내용 | 계량 | 게이트 | 재현성 |
|---|---|---:|---:|---:|
| 현재 | — | 35 | 55 | 60 |
| P0 | M0 실측 + `gate:pdf` 수리 + S1/S2 착지 + 환경 동등성 게이트 | 35 | 75 | 80 |
| P1 | 라벨 공장 가동, 첫 5장 IND-2 출고 | 60 | 80 | 85 |
| P2 | 15장·n 충족, 8지표 첫 산출 + 3회 반복 + 서명 | 85 | 85 | 90 |
| P3 | IND-3 판정 소진 + 공급자 매트릭스 + live-gates required 승격 | **90+** | **90+** | 90 |

P3 시점 종합(기존 강점 축 유지 가정): 아키텍처 88 · 안전 92 · 관통 85(15장
관통으로 상승) · 계량 90 · 게이트 90 · 재현성 90 → **종합 약 89~90**.

점수는 각 단계의 산출물이 존재할 때만 오른다. 설계 문서 자체는 0점이다.
