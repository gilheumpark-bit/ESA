실제 출력이 아니라 가정한 API를 상대로 작성됐고, 그중 LINE만 라이브에서 우연히
밟아 고쳐졌던 것이다. 나머지는 픽스처가 없어서 드러나지 않았다.

## 수리 요지

- **필드 수용 확대**: `position ?? startPoint`(TEXT), `position ?? center`(CIRCLE)
- **분류기 재작성**: 이름을 토큰으로 끊어 ① 토큰 완전일치 → ② 2글자+ 접두 → ③ 3글자+ 부분일치.
  1글자 키는 완전한 토큰일 때만 유효하다(`M-1`은 모터, `MCC-1`은 모터가 아니다).
  어휘 보강: `cubicle`→panel, `ats`→switch
- **좌표 정밀도 보존**: 정수 격자 폐기, 근접 끝점 병합은 허용반경 군집화로 이관.
  격자 크기를 좌표계에 고정하면 어떤 값을 골라도 어떤 축척에서는 틀린다
- **레이어 필터**: 치수·해칭·도면틀류 접두 제외, `ignoreLayers` 옵션으로 덮어쓰기 가능
- **근접 임계 재설계**: bbox 대각선 비율 → **심볼 간 최근접 거리 중앙값의 60%**.
  세로로 긴 단선도에서 대각선 3%는 6단위라 옆 15단위 텍스트가 전부 미결합됐다.
  도면 전체 크기가 아니라 심볼이 놓인 간격이 텍스트 거리를 결정한다
- **행 정렬 인출**: 정격표를 옆 열에 몰아 쓰는 실무 관례 수용. 단 **높이가 겹치는
  심볼이 정확히 하나일 때만** — 후보가 둘이면 귀속을 지어내지 않고 포기한다
- **파싱 실패 흡수**: 손상·비형식 입력은 예외가 아니라 confidence 0 결과 (DXF·PDF 동일 계약)

## 이 100%를 어디까지 믿을 것인가

**합성 도면에 한해서만 유효하다.** 근거와 한계를 같이 적는다.

믿을 수 있는 근거 — 정답이 파서 출력이 아니라 작도 선언에서 파생됐고(닫힌 순환 아님),
같은 픽스처가 수리 전에는 실제로 실패했다(20%·9.5%·0% 행이 남아 있다).

믿으면 안 되는 이유 — 임계값 두 개(간격 60%·행 밴드 70%)를 내가 이 15장을 보며
골랐다. 끼워맞춤 위험이 있다. 그래서 **조정에 쓰지 않은 적대 케이스 19건**을 따로
붙였다(`dxf-parser-adversarial.test.ts`): 두 심볼 정중앙 텍스트는 어디에도 붙지 않을 것,
배율 6종(0.001~100000) 불변, 어휘 밖 이름은 조용히 부하로, 빈/비DXF 입력, 0길이 선분,
심볼 1개 도면. 이 중 2건이 실제로 실패해 결함 6번을 잡아냈다.

그럼에도 **실제 현장 도면 검증은 여전히 0장**이다. 합성 도면은 내가 아는 방식으로만
작성돼 있고, 실무 도면은 내가 모르는 방식으로 작성돼 있다. 단계 벡터의 V(실증)는
이번 작업으로 올라가지 않았다 — 올라간 것은 I(구현)와 회귀 차단이다.

## 알려진 한계 (선언)

- **레이어 접두 과잉 차단**: `GRID_POWER`처럼 `grid`로 시작하는 전력 레이어도 함께
  빠진다. 테스트로 못박아 뒀고(`적대 — 레이어 필터 과잉 차단`), 실도면에서 만나면
  `ignoreLayers` 옵션으로 덮는다.
- **중첩 블록 내부 심볼**: 모델스페이스만 순회하므로 큐비클 블록 안의 개별 차단기는
  1개 노드로 뭉친다. 현재는 이것이 의도된 동작이며(큐비클 1면 = 노드 1개),
  내부 분해가 필요하면 별도 옵션이 필요하다.
- **PDF 벡터 경로**: 좌표 정밀도·파싱 실패 계약은 DXF와 맞췄으나 픽스처가 없다.
  PDF 픽스처 생성은 별건이다.
- **스캔 래스터**: VLM 경로이며 이번 범위 밖.
